# macOS UI Shell v2.1（阶段详情冻结）

## 核心定义

- 平台是软件全流程托管交付系统，不是 IDE。
- 阶段详情页是执行控制页，不是阶段介绍页。
- 首屏只回答一个问题：此刻要做什么决策。

## 侧栏收缩原则

- 侧栏宽度只服务：品牌识别、导航项、用户入口。
- 宽度按导航最长文案收紧，不允许解释性文案撑宽。
- 品牌区仅保留产品名，不保留“系统说明”副标题。

## 阶段详情固定骨架

1. 决策条（首屏）
2. 生命周期轨道（单一表达）
3. 主工作区 + 辅助区

### 决策条字段约束

- 仅保留：`阶段`、`状态`、`更新时间`、`唯一决策问题`、`主动作 + 最多 2 个次动作`
- 不出现：执行主体、优先级、项目总进度条、全局刷新状态、daemon/协议/socket、生命周期吞吐、系统健康、项目库筛选

### 决策问题（按阶段固定）

- 立项：`是否确认立项并进入 PRD`
- PRD：`是否冻结需求边界`
- UI：`是否确认方案可进入研发`
- 研发：`继续自动推进还是人工介入`
- 测试：`是否达到发布门槛`
- 发布：`是否执行发布或回滚`
- 维护：`记录问题继续维护，还是触发新一轮立项`

## 生命周期轨道

- 只承担阶段切换导航与阶段位置表达。
- 不增加说明性文案，不出现重复阶段表达。

## 主工作区（每阶段最多 3 模块）

### 立项

- `项目简介`
- `可行性报告`
- `参考资料与关键判断`

### PRD

- `范围与场景`
- `功能与验收`
- `版本状态`

### UI

- `页面结构`
- `关键交互`
- `待确认设计点`

### 研发

- `执行面`
- `依赖与阻塞`
- `当前产出状态`
- `前后端需求分析`
- `任务拆分`
- `Git/GitHub 协作`
- `稳定预览`
- `编码 -> CodeReview -> 修复 -> 测试` 循环
- `交付归档`

### 测试

- `测试结论`
- `失败与阻塞`
- `回归状态`

### 发布

- `发布准备`
- `发布状态`
- `回滚条件`

### 维护

- `运行反馈`
- `风险信号`
- `下一轮建议`

规则：

- 模块无内容直接隐藏，不显示占位符。
- 不为了“统一”而强行让阶段结构一致。

## 辅助区固定化

- 固定两个卡片：`风险与阻塞`、`事件流`
- `事件流`只接受实时事件（`detail.events`），不渲染蓝图兜底。
- 风险优先展示阶段实时风险与阻塞信息。

## 下载分级与阶段策略

下载分三类：

- `原始输入资料`
- `阶段产物快照`
- `审计留档文件`

阶段策略：

- 立项：`可行性报告` + `参考资料原件`
- PRD：`PRD 快照`
- 测试：`测试报告` + `验收结论快照`
- 发布：`发布记录/发布包` + `回滚方案留档`
- UI：可选 `UI 方案快照`（在线查看优先）
- 维护：可选 `维护记录`（在线查看优先）
- 研发：默认展示任务拆分、协作记录、稳定预览与交付归档清单；无文件时显示待生成或只读状态

交互规则：

- 有文件：显示可执行入口（下载/查看）
- 无文件：显示 `待生成` 或在线查看状态，不制造空壳主动作

## 后端契约（阶段详情）

### StageDetail

- `projectId`
- `stage`
- `status`
- `updatedAt`
- `objective`
- `inputContexts[]`
- `outputArtifacts[]`
- `downloads[]`
- `riskItems[]`
- `blockerReason?`
- `events[]`（实时）
- `primaryAction`
- `secondaryActions[]`（最多 2）
- `downloads[]`

### StageDetail.downloads[]

- `id`
- `title`
- `category`：`rawInput | stageSnapshot | auditArchive`
- `availability`：`ready | pending | viewOnly`
- `filePath?`
- `updatedAt?`
- `contentType?`

## 当前实现说明（v2.1）

- 阶段详情 UI 已按“决策条 -> 轨道 -> 工作区/辅助区”重排。
- 阶段工作区按阶段差异化渲染，控制在 2-3 个模块。
- 下载入口使用结构化渲染（以 `stage.downloads[]` 为目标契约）。
- 事件流只显示实时 `detail.events`。
